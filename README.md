
## How to build

```sh
npm run build
```

## Configuration

Create a `.env` file in the project root:

```
MINA_RPC_NETWORK_URL=
MINA_SENDER_PRIVATE_KEY=
MINA_TX_FEE=
MINA_NETWORK=

NORI_MOCK_ADMIN_PRIVATE_KEY=
NORI_MOCK_ADMIN_ADDRESS=
NORI_MOCK_TOKEN_PRIVATE_KEY=
NORI_MOCK_TOKEN_ADDRESS=
NORI_MOCK_ADMIN_PUBLIC_KEY=
NORI_MOCK_TOKEN_ID=
NORI_MOCK_TOKEN_ALLOW_VK_UPDATE=
```

- **MINA_RPC_NETWORK_URL**: Mina network RPC endpoint URL.
- **MINA_SENDER_PRIVATE_KEY**: private key of the transaction sender.
- **MINA_TX_FEE**: transaction fee (e.g. `0.1`). Defaults to `0.1` if not set.
- **MINA_NETWORK**: target network (`mainnet`, `devnet`, `lightnet`).

- **NORI_MOCK_ADMIN_PRIVATE_KEY**: private key for the mock NoriTokenBridge admin account. Generated by `npm run deploy` and written to `.env.nori-mock-token`. Must **not** be set when running `npm run deploy` — it will be rejected.
- **NORI_MOCK_ADMIN_ADDRESS**: deployed address of the mock NoriTokenBridge contract. Generated by `npm run deploy`.
- **NORI_MOCK_TOKEN_PRIVATE_KEY**: private key for the FungibleToken account. Generated by `npm run deploy`. Must **not** be set when running `npm run deploy` — it will be rejected.
- **NORI_MOCK_TOKEN_ADDRESS**: deployed address of the FungibleToken contract. Generated by `npm run deploy`.
- **NORI_MOCK_ADMIN_PUBLIC_KEY**: public key of the contract admin account. Generated by `npm run deploy` (defaults to the public key derived from `MINA_SENDER_PRIVATE_KEY`).
- **NORI_MOCK_TOKEN_ID**: token ID of the FungibleToken. Generated by `npm run deploy`.
- **NORI_MOCK_TOKEN_ALLOW_VK_UPDATE**: controls whether the FungibleToken verification key can be updated (`true`/`false`).

## How to bake integrity hashes

When `NoriTokenBridge` or `FungibleToken` are modified, recompile and update the integrity data. Changes to either contract require re-running `bake-vk-hashes` before running [`deploy`](#how-to-deploy), [`migrate-vk-to-tag`](#how-to-update-the-verification-key), or [`update-vk`](#how-to-update-the-verification-key-directly-not-recommended).

Run:

```bash
npm run bake-vk-hashes
```

This regenerates:

```
src/integrity/NoriTokenBridge.VkHash.json    — verification key hash
src/integrity/NoriTokenBridge.VkData.json    — verification key data
src/integrity/FungibleToken.VkHash.json
src/integrity/FungibleToken.VkData.json
```

These files are checked at runtime during:

- `npm run deploy`
- `npm run update-vk` / `npm run migrate-vk-to-tag`

If the compiled verification key does not match the stored integrity hash, these commands will throw before any transaction is submitted.

For `migrate-vk-to-tag` the relationship with `bake-vk-hashes` is more nuanced: `migrate-vk-to-tag` is a VK migration workflow that runs `bake-vk-hashes` on the target commitish rather than the current checkout as part of its process — see [How to update the verification key](#how-to-update-the-verification-key).

## How to deploy

Ensure your `.env` contains:

```
MINA_RPC_NETWORK_URL=<url>
MINA_NETWORK=<mainnet|devnet|lightnet>
MINA_SENDER_PRIVATE_KEY=<your-private-key>
MINA_TX_FEE=0.1
```

Do **not** set `NORI_MOCK_ADMIN_PRIVATE_KEY` or `NORI_MOCK_TOKEN_PRIVATE_KEY` — the deploy script generates fresh key pairs and will reject the run if either is already set.

Clear your o1js cache before deploying:

```bash
rm -rf ~/.cache/o1js/
```

Run:

```bash
npm run deploy [adminPublicKeyBase58]
```

- `[adminPublicKeyBase58]`: optional. The public key of the account with admin permissions over the contract (`setVerificationKey`). If omitted, defaults to the public key derived from `MINA_SENDER_PRIVATE_KEY`.

After deploy, a `.env.nori-mock-token` file is created in the root directory containing:

```
NORI_MOCK_ADMIN_PRIVATE_KEY=...
NORI_MOCK_ADMIN_ADDRESS=...
NORI_MOCK_TOKEN_PRIVATE_KEY=...
NORI_MOCK_TOKEN_ADDRESS=...
NORI_MOCK_ADMIN_PUBLIC_KEY=...
NORI_MOCK_TOKEN_ID=...
NORI_MOCK_TOKEN_ALLOW_VK_UPDATE=true
```

Copy these values into your `.env` file.

## How to burn

The burn script mints tokens to the sender and then immediately burns them, emitting a `BurnEvent`. This is what the Rust worker detects.

Ensure your `.env` contains:

```
MINA_RPC_NETWORK_URL=<url>
MINA_NETWORK=<mainnet|devnet|lightnet>
MINA_SENDER_PRIVATE_KEY=<admin-private-key>
NORI_MOCK_TOKEN_ADDRESS=<deployed-token-address>
MINA_TX_FEE=0.1
```

Run:

```bash
npm run burn <amount>
```

- `<amount>`: token amount in base units (e.g. `1000000` for 1 token at 6 decimals).

To burn repeatedly in a loop:

```bash
npm run burn-with-a-furious-vengeance
```

## How to update the verification key

`setVerificationKey` is a provable contract method: the proof must be generated using the circuit that is currently deployed on-chain. This means you must be running the code from the currently deployed contract version — not the version you are migrating to. The new verification key data is sourced separately from the target release's committed integrity files.

The recommended approach is `migrate-vk-to-tag`, which handles the full workflow automatically.

**Before you do anything else**, check out the git tag that corresponds to the currently deployed contract version:

```bash
git checkout <currently-deployed-tag>
```

This is the tag whose circuit is live on-chain. If you run `migrate-vk-to-tag` from the wrong checkout, the proof will be generated by the wrong circuit and will be rejected by the contract.

Once checked out to the deployed tag:

1. Reinstall dependencies from scratch and build:

```bash
npm ci && npm run build
```

2. Clear the o1js cache:

```bash
rm -rf ~/.cache/o1js/
```

3. Ensure your `.env` contains:

```
MINA_RPC_NETWORK_URL=<url>
MINA_NETWORK=<mainnet|devnet|lightnet>
MINA_SENDER_PRIVATE_KEY=<contract-admin-private-key>
NORI_MINA_TOKEN_BRIDGE_PRIVATE_KEY=<deployed-bridge-private-key>
MINA_TX_FEE=0.1
```

`NORI_MINA_TOKEN_BRIDGE_PRIVATE_KEY` must be the private key of the deployed NoriTokenBridge account — the contract address is derived from it. `MINA_SENDER_PRIVATE_KEY` must be the contract admin's private key.

4. Run:

```bash
npm run migrate-vk-to-tag <targetTagOrCommitSHA>
```

where `<targetTagOrCommitSHA>` is the release you are migrating **to**.

The script clones the target commitish to a temporary directory, installs dependencies, runs `bake-vk-hashes`, verifies the committed integrity files are not stale, then invokes `update-vk` against the target integrity files using the current checkout's circuit to generate the proof. The temporary directory is cleaned up on completion or failure.

## How to update the verification key directly (not recommended)

> **Warning:** `update-vk` bypasses the integrity verification step performed by `migrate-vk-to-tag`. If the integrity files you supply are stale or incorrect, the contract will be bricked. Only use this if you have independently verified the integrity files are correct and understand the risks.

You must still be checked out to the currently deployed contract version. Check out the deployed tag, then:

```bash
git checkout <currently-deployed-tag>
npm ci && npm run build
rm -rf ~/.cache/o1js/
```

Ensure your `.env` contains:

```
MINA_RPC_NETWORK_URL=<url>
MINA_NETWORK=<mainnet|devnet|lightnet>
MINA_SENDER_PRIVATE_KEY=<contract-admin-private-key>
NORI_MINA_TOKEN_BRIDGE_PRIVATE_KEY=<deployed-bridge-private-key>
MINA_TX_FEE=0.1
```

Then run:

```bash
npm run update-vk -- <path/to/NoriTokenBridge.VkData.json> <path/to/NoriTokenBridge.VkHash.json>
```
